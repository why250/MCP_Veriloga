Verilog-A in ADS Design Kits

This topic describes the steps necessary to distribute Verilog-A models in an ADS Design Kit either as Verilog-A source or as a set of compiled libraries. Design Kits provide a convenient way to distribute Verilog-A models and the steps are very similar to the steps to create conventional Design Kits.

For an overview of the process and the specific steps involved in creating an ADS Design Kit, refer to the Design Kit Development documentation.

Verilog-A modules in a Design Kit are automatically made available to workspaces using that design kit. Verilog-A files must be placed in the design kit veriloga directory: 

design_kit/veriloga

If the Verilog-A source will be shipped in the kit then this is all that needs to be done since the design kit Verilog-A directory is in the simulators Verilog-A auto-compile path. When a simulation is run all Verilog-A files in the auto-compile path are compiled if required and then loaded into the system. The auto-compile path is specified in hpeesofsim.cfg. When a design kit is made part of a project ADS automatically creates a config file with the design kit Verilog-A directory added to the auto-compile path.

With factory settings the following directories are in the path: 

CURRENT_PROJECT/veriloga
DESIGN_KIT/veriloga
$HOME/hpeesof/veriloga
$HPEESOF_DIR/custom/veriloga
$HPEESOF_DIR/veriloga

Each Verilog-A file in each directory is compiled; the file must have a .va extension. A compilation error in any directory will stop the simulation. The path is searched from the top. Modules with the same name higher in the path hide those lower in the path. What's important for design kits is to understand that if the user places a module in the Verilog-A directory of their current project and it has the same name as one in a design kit then the users module will be used over a module of the same name in the design kit.

Environment Setup to Run Verilog-A Compiler

The Verilog-A compiler, vamscomp, is run from the command line. If the system is already configured to run ADS tools from the command line then the following setup may be ignored. Below we assume an ADS installation directory when setting HPEESOF_DIR.

Linux (bash syntax) 
export HPEESOF_DIR=<ADS_Install_DIR>
export TIBURON_HOME=$HPEESOF_DIR/tiburonda
export PATH=$HPEESOF_DIR/bin:$PATH
export LD_LIBRARY_PATH=$HPEESOF_DIR/lib/linux_x86_64/:$LD_LIBRARY_PATH
Windows 
set HPEESOF_DIR=<ADS_Install_DIR>
set TIBURON_HOME=%HPEESOF_DIR%\tiburonda
set PATH=%HPEESOF_DIR%\bin;%PATH%

Protecting IP by Distributing Verilog-A in Compiled Form

When sharing a Verilog-A model with a third party, the model may be provided in compiled form only with no Verilog-A source provided. The compiled model is instantiated and simulated in exactly the same way as a model provided with source. When the system is processing a directory in the Verilog-A auto compile path, if it detects the directory contains pre-compiled modules it will load these modules into the system. A directory is identified as being configured to load pre-compiled modules when the system sees a design database called work.db in a directory named with the current Tiburon-DA version number. For example for ADS 2019 this is, 

2.91.101/work.db

If this file is present then the system assumes this is a valid compiled Verilog-A directory. It will open the database and load all pre-compiled Verilog-A modules into the system making them ready for simulation. If the database load fails then this is a fatal error, the system will stop the simulation. Any VA files in the directory are ignored.

Creating a compiled database and platform libraries

The Verilog-A compiler, vamscomp, is used to create a compiled database. One or more files may be compiled into a compiled database; there is always one database per directory. 

# Go to the directory of interest
>cd design_kit/veriloga 
# Remove any existing compiled database
>rm -rf 2.91.101
# Run the compiler                   
>vamscomp -distribute -amswork . file.va file2.va file3.va
Keysight Technologies Verilog-A Compiler Version 2.91.101.041818
Copyright 2018, Keysight Technologies.
Making the database ready for distribution.

Options:

  • -distribute option to strip all sources and encrypt any remaining information required for simulation.
  • -amswork option to name the output directory. For ADS design kits, if we are in the kit Verilog-A directory then we need the db directly in this directory and so the argument to this option is always ".".
  • the list of sources to compile
  • you must completely delete the compiled database directory before running the compiler.
  • The distribute option must be used. It strips all IP out of the database and encrypts all remaining information. The only information available in clear text in the database after calling -distribute are the generic Verilog-A natures and disciplines. Once the -distribute option is used the compiler will no longer run successfully in this directory.

The original Verilog-A source may now be removed. When the simulator is processing this directory it will find the database, open it, and load the compiled modules making them available for simulation.

Compiled Verilog-A Directory Structure

Inspecting the veriloga directory, the module database and the compiled module library for this directory will be available. These are the only files required. Any other log files may be deleted. When supporting a single ADS release, with a single platform, only two files will be present. All others may be deleted. 

veriloga/2.91.101/work.db
veriloga/2.91.101/analog_files/lib.linux_x86_64/lib_tmc.so

If both Linux and Windows are compiled then we will see the following structure; with one database file used on all platforms and one shared library per platform, typically one for Linux 64 and one for Windows 64. 

veriloga/2.91.101/work.db
veriloga/2.91.101/analog_files/lib.linux_x86_64/lib_tmc.so
veriloga/2.91.101/analog_files/lib.win64/lib_tmc.dll

If multiple ADS releases are supported then there will be a directory per ADS release as follows 

veriloga/2.80.910/...  # 2014_01
veriloga/2.90.320/...  # 2015_01
veriloga/2.91.101/...  # 2019

When ADS boots it will look in the associated version directory to load the compiled models from that directory. Note that at the time of writing the release version for 2019 is 2.91.101. This may change in the final release.

Supporting Multiple Platforms

The compiler must be run and platform-specific compiled libraries created for each supported platform. Compiles must always be done starting in a clean directory. The procedure to follow is as follows

  1. Run the compiler on one platform, creating the database and the shared library for this platform, if we ran on Linux we will see the following two files 
    ...2.91.101/work.db
...2.91.101/analog_files/lib.linux_x86_64/lib_tmc.so
     

  2. Copy the source to another directory and do the same compile on win64

  3. Copy the win64 DLL: 
    ...2.91.101/analog_files/lib.win64/lib_tmc.dll
  4. Once complete we will have 3 files in total used to support all ADS platforms; the database which is platform independent and shared libraries, one per platform. 
    ...2.91.101/work.db
...2.91.101/analog_files/lib.linux_x86_64/lib_tmc.so
    ...2.91.101/analog_files/lib.win64/lib_tmc.dll

Final Checks Before Distributing the Compiled Kit

  1. Compiled Verilog-A directories should contain no Verilog-A source files; check that only the design database and the shared libraries are present

  2. The compiler -distribute option cannot be omitted otherwise IP will exist in the database in clear text. Verify that there is no IP in the database by dumping and visually inspecting the file 

    sqlite3 ...veriloga/2.91.101/work.db .dump


    It should contain only generic discipline information and encrypted fragments.

Protecting IP by encrypting Verilog-A code

Verilog-A code can be encrypted through -encrypt option of vamscomp tool. The encrypted code is simulated in the usual way.

  • Verilog-A file encrypted in versions prior to ADS 2019 cannot be used in ADS 2019 or later.
  • When -encrypt is used with other vamscomp options, only -encrypt option is used and other options are ignored.


Call the encryption tool on a Verilog-A file 

>vamscomp -encrypt test.va
Keysight Technologies Verilog-A Compiler Version 2.91.101.041818
Copyright 2018, Keysight Technologies.
Created verilog encrypted file "test.vax"

The .vax file may be placed in the design kit Verilog-A directory and the original Verilog-A may be deleted.

When the Verilog-A code contains an analog block then generated C representing parts of the model will exist on disk in the Verilog-A object cache. This code is compiled by gcc and then deleted. Compilation only happens if the contents of the file are uniquely different than existing compiled modules available in the Verilog-A object cache. The C code does not contain the original Verilog-A source; however, it is derived from that source. This should be considered when using encryption to share IP.

Once encrypted the .vax file will run in all future versions of ADS, GoldenGate and any simulator integrating the Keysight Verilog-A simulation component.
